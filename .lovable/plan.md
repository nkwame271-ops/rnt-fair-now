## Goal
Put a paywall in front of **Report Safety Issue** (tenant / landlord / student). Emergency Panic alerts stay free. Fees are routed through the existing escrow + split engine, with a student-specific split that includes NUGS and CM.

## Scope
- Affects only the Safety Report flow (`src/pages/shared/ReportSafetyIssue.tsx` + role wrappers).
- `SafetyPanicButton` / `submit-safety-report` for `report_kind = 'panic_emergency'` is **unchanged** (must remain free).

## Data model (migration)
1. New table `pending_safety_report_drafts` — mirrors `pending_complaint_drafts`:
   - `id`, `user_id`, `user_role` (`tenant|landlord|student`), `payload jsonb`, `evidence_paths text[]`, `amount numeric`, `status` (`pending_payment|paid|materialized|expired`), `reference text`, `materialized_report_id uuid`, `expires_at`, timestamps.
   - RLS: owner can manage own; service role full access.
2. Seed two new `payment_type`s in `split_configurations`:
   - `safety_report_fee` → recipients `rent_control` (IGF), `admin`, `platform` (standard 3-way split, flat GHS amounts to be set in Engine Room).
   - `student_safety_report_fee` → recipients `igf`, `nugs`, `platform`, `cm` (student 4-way split, matches existing `student_complaint_fee` shape).
3. Seed defaults into `fee_catalog`-equivalent rows the Engine Room reads (same rows the existing `student_complaint_fee` uses); amounts default to a placeholder (e.g. GHS 5) — Super Admin edits in Engine Room.

## Backend
4. `supabase/functions/paystack-checkout/index.ts`: add two new `type` branches mirroring `student_complaint_draft`:
   - `safety_report_draft` → resolves amount from `split_configurations` where `payment_type='safety_report_fee'`, stores `draftId` in metadata, sets `escrow_transactions.payment_type='safety_report_fee'`.
   - `student_safety_report_draft` → same, with `payment_type='student_safety_report_fee'`.
5. `supabase/functions/_shared/finalize-payment.ts`: on success for these types, read draft, insert into `safety_reports` (reusing the columns `submit-safety-report` writes), run the same fan-out (audit log, SMS to safety contacts, admin notifications, EmergencyAlertRinger realtime channel is unaffected because that's panic-only). Mark draft `materialized`. Receipts generated by existing receipt pipeline (no extra work — receipts already emit for any successful escrow_transaction).

## Frontend
6. `src/pages/shared/ReportSafetyIssue.tsx`:
   - Keep the existing form. On **Submit**:
     - Upload evidence to `safety-evidence` storage (as today).
     - Insert a row in `pending_safety_report_drafts` with the full payload (category, severity, description, hostel, school, coords, evidence_urls, role, silent flag).
     - Call `paystack-checkout` with `type = role === 'student' ? 'student_safety_report_draft' : 'safety_report_draft'` and `draftId`.
     - Redirect to `authorization_url`.
   - Add a small "Safety Report Fee" notice above the Submit button showing the current fee (read from `split_configurations` sum for the matching `payment_type`).
   - On payment return (existing `/payment/callback` handler), success toast + redirect to `MySafetyReports`.
7. `SafetyPanicButton` and the `panic_emergency` path in `submit-safety-report` — **no changes** (free).

## Engine Room / Super Admin
8. The Engine Room editor (`src/pages/regulator/EngineRoom.tsx`) already lists all `payment_type`s found in `split_configurations`. Once seeded, "Safety Report Fee" and "Student Safety Report Fee" appear automatically. Add friendly labels in the same label map the page uses for `student_complaint_fee`.
9. `StudentRevenue.tsx` dashboard already aggregates by `payment_type IN (student_*)`. Extend its filter list to include `student_safety_report_fee` so the new revenue surfaces in the Student Revenue dashboard with the IGF / NUGS / Platform / CM breakdown.

## Rules enforced
- Panic emergency = free (unchanged code path).
- Safety report = paid; report row only created after `finalize-payment` succeeds.
- Split logic chosen by user_role in the frontend `type` selection and enforced server-side.
- Receipts + escrow visibility follow existing rules for the chosen `payment_type`.

## Out of scope
- No changes to Panic Alert ringer, KYC gates, or notification permission flow.
- No changes to the receipt template itself.
